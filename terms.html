Slang & Shorthand
ACK Acknowledged
Common in Slack messages. e.g. "ACK" as a quick reply to confirm you've seen or noted something. Funny story, I got super freaked out cuz I thought my manager was mad at me when I received this for the first time 😅
FFT Feel Free To
Common in Slack messages. e.g. "FFT include me in that discussion."
SFDC Salesforce
Standard industry shorthand for Salesforce (SalesForce Dot Com). Not Datadog-specific — you'll see it everywhere.
Skip Manager
Your manager's manager.
BYO Bring Your Own
In context, "bring your own" agent — a customer builds and hosts their own AI agent instead of using Datadog's Bits AI.
Roles
EM Engineering Manager
Manages a team of engineers — responsible for hiring, performance, and team health. Different from a PM who owns the product direction.
TPM Technical Program Manager
Coordinates and drives execution across engineering teams, managing timelines, dependencies, and cross-team alignment. Less focused on what to build (that's the PM) and more focused on making sure it gets built on time.
PMM Product Marketing Manager
Handles positioning, messaging, and go-to-market strategy for a product. Bridges product and sales/marketing — often owns launch plans, pricing narratives, and customer-facing content.
Engineering & Code
PR Pull Request
How engineers propose changes to code. Instead of pushing directly to the main codebase, they open a PR — essentially saying "here are my changes, can someone review before we merge?" Other engineers review, leave comments, and approve before it goes in.
Code Review
When another engineer reads through the changes in a PR before they get merged. They're checking for bugs, bad logic, security issues, or messy code. It's a quality control step — most companies require at least one approval before a PR can be merged.
FF Feature Flag
A way to turn a feature on or off without deploying new code. Lets teams roll out features gradually or test them with a subset of users.
Endpoint
A specific URL in an API that does one thing. For example, /logs/search retrieves logs, /metrics/query queries metrics. Each endpoint is like a door into the product — you call it with the right inputs and get a specific output back.
gRPC
A way for services to communicate with each other, similar to a REST API but faster and more efficient. Instead of sending human-readable text (like JSON), it sends compact binary data. Relevant at Datadog because some API calls go over gRPC rather than HTTP, which creates gaps in metering — if you can only meter HTTP traffic, gRPC calls slip through uncounted.
UUID Universally Unique Identifier
A randomly generated ID used to uniquely identify something (like a user) without storing any real personal information. Instead of storing a user's email directly in a data system, you store their UUID. The email lives in one place (the user profile), and the UUID just points to it.
This matters for privacy. If a user requests to be forgotten (right to be forgotten / GDPR), you just sever the connection between the UUID and their profile — rather than having to hunt down and delete their email across every data system that ever touched it.
Namespace
A container that groups related things (functions, variables, classes) under a unique name to avoid conflicts. Two teams can both have a function called get_user — as long as they're in different namespaces, they don't clash. Typical syntax: team1.get_user vs team2.get_user.
.yaml YAML file
"YAML Ain't Markup Language" — a recursive acronym (a bit of a programming joke). A file format used for configuration. Human-readable and structured with indentation. Very common in DevOps and infrastructure tooling — you'd see it in Kubernetes configs, CI/CD pipelines, etc.
OpenAPI
A standard file format (usually YAML or JSON) that formally describes what an API can do — what endpoints exist, what inputs they take, what they return, and how to authenticate. It acts as a blueprint for the API.
At Datadog, engineers write and maintain OpenAPI specs alongside the code. When an endpoint is added or changed, the spec is updated too. Those files feed into the public API docs site, where a tool renders them into the human-readable documentation developers use. The key thing: OpenAPI is machine-readable — tools, AI agents, and client libraries can read it and automatically understand how to interact with the API.
Types of Files
  • Config files — store settings and parameters for an app (e.g. database URL, port number, feature flags). Examples: .yaml, .json, .toml, .env
  • Data files — store structured data. Examples: .csv, .json, .xml
  • Source code files — the actual code. Examples: .py (Python), .js (JavaScript), .go (Go)
  • Markup files — define structure/formatting for content. Examples: .html (web pages), .md (Markdown)
  • Binary files — not human-readable, compiled for machines. Examples: .exe, .bin
  • Log files — records of what an app did, used for debugging. Usually .log or .txt
  • Lock files — record exact dependency versions so everyone installs the same thing. Examples: package-lock.json, Pipfile.lock
  • Schema files — define the structure/shape of data. Examples: .sql, .graphql
Product & Business
SKU Stock Keeping Unit
In retail, a SKU is a unique identifier for a product variant (like different shirt sizes). At Datadog, each product or product group that gets priced differently is a different SKU.
TCO Total Cost of Ownership
The full cost of using something — not just the purchase price but all associated costs (setup, maintenance, operations, etc.). Used in the pricing project to compare the total cost of using Bits AI vs. building your own agent solution.
Product Brief / PRD
At Datadog, Product Brief and PRD (Product Requirements Document) are the same thing. The goal is to get the important "why and what" across in an organized manner. Every PM writes them a bit differently — read examples early and develop your own style.
PFD Product Field Day
An internal Datadog event where PMs present their products and roadmaps to the sales and field teams (AEs, SEs, customer success). The goal is to give the people who talk to customers every day the context they need to have informed conversations about what's being built and when.
Arbitrage
Exploiting a price difference between two things that are effectively the same to get a cheaper deal. In Datadog's pricing context: if MCP calls cost money but API calls were free, customers could get the exact same data by routing through the API instead — same result, zero cost. That's arbitrage. Metering at the API layer prevents this.
Critical Path
The sequence of tasks that determines the earliest possible finish date for the whole project. If anything on the critical path is delayed, the entire project is delayed.
Example: in the API Visibility project, metering infrastructure (M1) has to be done before the dashboard (M2) can launch — so M1 is on the critical path. A slip in M1 pushes everything else back.
API-first vs UI-first
  • UI-first — you build the visual interface first and the API just serves that UI. The API becomes tightly coupled to what the UI needs and hard for others to use independently.
  • API-first — you design the API as the core product. The UI is just one consumer of it, same as any external developer or agent. Everything is built to be programmable from the start.
At Datadog this matters because customers aren't just clicking dashboards — they're running agents, scripts, and automations that hit the API directly. If the API is an afterthought, those use cases break down.
Datadog Tech
SDK Software Development Kit
A package of tools and libraries that developers use to interact with a product programmatically. At Datadog, SDKs let customers write code to send data to or pull data from Datadog without making raw API calls themselves.
API, CLI, and MCP
These are different surfaces — different ways a user or program can interact with Datadog.
  • API (Application Programming Interface) — the direct, raw way to retrieve data from Datadog via HTTP requests. Everything else is built on top of this.
  • CLI (Command Line Interface) — a tool you run in your terminal. Wraps API calls into human-friendly commands like pup logs list.
  • MCP (Model Context Protocol) — a newer protocol that lets AI agents interact with Datadog programmatically.
All three ultimately resolve to the same underlying API call on the backend. Datadog's pricing model meters the underlying API call regardless of which surface triggered it — if one surface were cheaper, customers would just route through that one (arbitrage). Metering at the API layer makes the model arbitrage-proof.
